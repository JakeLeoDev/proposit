# i18n / Localization

This project uses [next-intl](https://next-intl.dev/) for internationalization. Supported locales: **en** (English) and **de** (German). Every user-facing string must be translated — no hardcoded text in components.

## File Structure

```
messages/
  en.json          # English translations
  de.json          # German translations
i18n.ts            # Request config (locale detection, message loading)
lib/i18n-routing.ts  # Routing config (locales, pathnames)
middleware.ts      # Locale prefix middleware
app/[locale]/layout.tsx  # NextIntlClientProvider wraps all pages
```

## Translation Namespaces

Each top-level key in `messages/*.json` is a namespace:

| Namespace         | Scope                                                             |
| ----------------- | ----------------------------------------------------------------- |
| `navigation`      | Sidebar, navbar, breadcrumb segment labels                        |
| `auth`            | Login, signup, password reset, invitations                        |
| `dashboard`       | Dashboard welcome, statistics, overview                           |
| `proposals`       | Proposal CRUD, status management, collections                     |
| `qualifications`  | Qualification CRUD                                                |
| `certificates`    | Certificate CRUD                                                  |
| `attachments`     | Attachment CRUD                                                   |
| `categories`      | Category CRUD                                                     |
| `companies`       | Company CRUD                                                      |
| `persons`         | Contact person CRUD                                               |
| `products`        | Product collection and item CRUD                                  |
| `common`          | Shared labels, generic CRUD messages, UI defaults                 |
| `viewer`          | Proposal viewer (status badges, title page, signature, PDF, etc.) |
| `editor`          | Lexical editor toolbar, table actions, color picker               |
| `chat`            | AI chat interface                                                 |
| `ai`              | AI assistant context prompts                                      |
| `settings`        | User and organisation settings                                    |
| `crud`            | Reusable CRUD component labels (modal, list)                      |
| `promptTemplates` | Prompt template management                                        |
| `greeting`        | Greeting component                                                |

## How to Use Translations

### Client Components (`'use client'`)

```tsx
import { useTranslations } from 'next-intl';

export function MyComponent() {
	const t = useTranslations('proposals');
	return <h1>{t('title')}</h1>;
}
```

Multiple namespaces in one component:

```tsx
const t = useTranslations('proposals');
const tCommon = useTranslations('common');
const tViewer = useTranslations('viewer');
```

### Server Components

```tsx
import { getTranslations } from 'next-intl/server';

export default async function Page() {
	const t = await getTranslations('dashboard');
	return <h1>{t('title')}</h1>;
}
```

### Interpolation

```tsx
// messages/en.json: "greeting": "Hello, {name}!"
t('greeting', { name: 'Alice' });

// messages/en.json: "count": "{count} items"
t('count', { count: 5 });
```

## Rules for Adding New Strings

1. **Never hardcode user-facing text.** Every string visible to the user (labels, placeholders, toasts, error messages, dialog titles, button text, aria labels) must come from a translation function.

2. **Add keys to both locale files.** Every new key must be added to `messages/en.json` AND `messages/de.json` simultaneously. Missing keys in one file will cause the other locale to show raw keys or fall back.

3. **Choose the right namespace.** Place the key in the entity-specific namespace if it belongs to a single entity (e.g., `proposals.createdSuccessfully`). Use `common` only for truly shared strings (save, cancel, delete, generic errors).

4. **Use consistent key naming:**
   - Labels: `name`, `description`, `status`
   - Placeholders: `namePlaceholder`, `descriptionPlaceholder`
   - Success toasts: `createdSuccessfully`, `updatedSuccessfully`, `deletedSuccessfully`
   - Error toasts: `createFailed`, `updateFailed`, `deleteFailed`, `notFound`, `loadFailed`
   - Other: `selectCategory`, `searchProposals`, `noProposals`

5. **Service files and utility functions cannot use hooks.** If a non-component file needs translated text, either pass the translated string as an argument from the calling component or accept that it stays untranslated (e.g., internal error logs).

6. **Toasts always use translations.** Replace `toast.success('Created!')` with `toast.success(t('createdSuccessfully'))`.

7. **Form field placeholders use translations.** Replace `placeholder: 'Enter name'` with `placeholder: t('namePlaceholder')`.

8. **CRUD component defaults are localized.** `DataTable` and `CompactList` use `useTranslations('common')` for fallback defaults (search placeholder, empty message, button titles). If you pass explicit props, pass translated strings.

## Checklist for New Features

When adding a new feature or entity:

- [ ] Add all labels, placeholders, success/error messages to `messages/en.json`
- [ ] Add the corresponding German translations to `messages/de.json`
- [ ] Use `useTranslations` (client) or `getTranslations` (server) in every component
- [ ] Verify both locales render correctly by switching the URL prefix (`/en/...` vs `/de/...`)
- [ ] Check toast messages, dialog titles, empty states, and form placeholders
